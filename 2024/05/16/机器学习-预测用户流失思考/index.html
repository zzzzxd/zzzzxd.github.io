以下是一套标准的“从 0 到 1 解决流失率预测”的实战思考框架，分为 6 个阶段。

🗺️ 核心思维框架：CRISP-DM 改良版

定义问题 (Define) ➡️ 数据策略 (Data) ➡️ 特征工程 (Feature) ➡️ 建模选型 (Model) ➡️ 评估与解释 (Eval) ➡️ 业务落地 (Action)

第一阶段：定义问题 (Business Understanding)

⚠️ 最关键的一步：不要急着看数据，先问业务方（或假设）清楚目标。

  1. 什么是“流失” (Definition of Churn)?

    • 是用户注销账号？
    • 是连续 30 天未登录？
    • 是取消订阅（Subscription Cancellation）？
    • 是停止充值？
    • 决策：不同定义导致标签（Label）完全不同。通常定义为：在未来 $T$ 天内没有发生关键行为（如登录/购买）的用户。

  2. 预测的时间窗口 (Time Window) 是多少？

    • 我们要预测“下个月”会流失的人，还是“下周”？
    • 设定：观察窗口（Observation Window，如过去 3 个月的行为） + 缓冲期（Buffer，如 7 天） + 预测窗口（Prediction Window，如未来 30 天）。

  3. 业务目标是什么？(Objective)

    • 是为了召回（发优惠券）？ -> 关注 Recall（宁可错杀，不可放过）。
    • 是为了节省成本（只对高价值用户干预）？ -> 关注 Precision 或 ROI。
    • 是为了分析原因？ -> 关注 可解释性 (SHAP)。

第二阶段：数据策略 (Data Strategy)

🔍 没有数据怎么办？先盘点有什么，再决定怎么造。

  1. 数据源盘点：

    • 用户画像 (Demographics)：年龄、性别、地域、注册渠道、设备型号。
    • 行为日志 (Behavioral)：登录频率、停留时长、点击流、搜索记录、最后活跃时间 (Recency)。
    • 交易数据 (Transactional)：ARPU (平均收益)、充值次数、退款记录、优惠券使用情况。
    • 客服/反馈 (Support)：投诉次数、工单内容、满意度评分。

  2. 构建样本集 (Sampling & Labeling)：

    • 正负样本不平衡：流失用户通常是少数（可能只有 5%）。
    • 对策：需要采样（下采样多数类，或上采样少数类），或者在损失函数中调整权重 (scale_pos_weight)。
    • 时间穿越 (Data Leakage)：绝对禁止使用未来的数据！
      • 检查：特征必须严格发生在“观察窗口”内，标签必须严格发生在“预测窗口”内。

第三阶段：特征工程 (Feature Engineering)

🛠️ 这是提升模型效果的上限所在。对于流失率，有三类“杀手级”特征：

  1. RFM 模型变种：

    • R (Recency)：最近一次登录/购买距今天数（最强特征，越久越容易流失）。
    • F (Frequency)：过去 N 天的活跃频次/购买频次（趋势下降是危险信号）。
    • M (Monetary)：消费金额变化率。

  2. 趋势类特征 (Trend Features)：

    • 本周活跃度 vs 上周活跃度 vs 上月活跃度（计算斜率/变化率）。
    • 逻辑：一个从来不怎么活跃的人不叫流失，“突然不活跃了”才叫流失。

  3. 交互与统计特征：

    • 客诉次数 / 总订单数。
    • 优惠券核销率。
    • 社交关系密度（如果是有社交属性的产品，好友流失我也容易流失）。

  4. 处理缺失值与编码：

    • 类别特征（如城市）用 Target Encoding 或 Embedding。
    • 数值特征做归一化或分桶（Binning）。

第四阶段：建模选型 (Model Selection)

🤖 工业界首选：树模型。深度学习通常杀鸡用牛刀且难解释。

  1. Baseline 模型：

    • 逻辑回归 (Logistic Regression)：作为基准，提供可解释的系数，快速验证特征有效性。

  2. 主力模型 (SOTA)：

    • XGBoost / LightGBM / CatBoost。
    • 理由：
      • 对表格数据（Tabular Data）效果最好。
      • 能自动处理非线性关系。
      • 支持缺失值。
      • CatBoost 对类别特征处理极好，无需大量预处理。
      • 训练速度快，易于调参。

  3. 为什么不用深度学习 (Deep Learning)?

    • 除非你有海量的序列行为数据（如每秒的点击流），否则对于结构化表格数据，DL 往往不如 GBDT，且难以向业务方解释“为什么这个人要流失”。

第五阶段：评估与解释 (Evaluation & Interpretation)

📊 准确率 (Accuracy) 是没有意义的！因为 95% 的人都没流失，全猜“不流失”也有 95% 准确率。

  1. 核心指标：

    • AUC-ROC：衡量模型整体排序能力（区分好坏人的能力）。
    • Precision-Recall Curve (PR 曲线) & F1-Score：在不平衡数据下更真实。
    • Top-K Lift (提升度)：如果我们只干预预测概率最高的前 10% 用户，他们的流失率是随机抽取的多少倍？（业务最关心这个）。

  2. 可解释性 (Explainability)：

    • 业务方会问：“为什么你说他要流失？”
    • 使用 SHAP Values。
    • 输出：全局重要性（哪些特征最重要？通常是“最近未登录天数”）；局部解释（用户 A 流失是因为“投诉了 3 次”且“余额不足”）。

第六阶段：业务落地与闭环 (Deployment & Action)

🚀 模型上线不是结束，是开始。

  1. 分层运营策略：

    • 高风险 + 高价值：人工客服介入，送大额券。
    • 高风险 + 低价值：自动化短信/Push 召回，送小额券。
    • 低风险：不打扰，节省成本。

  2. A/B 测试 (Golden Standard)：

    • 实验组：根据模型预测进行干预。
    • 对照组：随机干预或不干预。
    • 核心指标：不仅看模型准不准，要看挽留成功率 (Retention Rate) 和 ROI（挽回的用户带来的利润 > 发的优惠券成本吗？）。

  3. 监控与迭代：

    • 监控 数据漂移 (Data Drift)：用户行为模式变了（比如疫情期间大家都不出门了），模型可能会失效，需要定期重训。

💡 面试/实战中的“加分项”话术

如果在面试中回答这个问题，说完上述流程后，补充以下几点会非常亮眼：

  1. “冷启动”问题：新用户没有行为数据怎么预测？
    • 答：利用注册渠道、设备指纹、初始引导行为做规则判断，或者单独建一个新用户流失模型。
  2. 因果推断 (Causal Inference)：
    • 答：预测出流失概率高，不代表发了券就能留住。有些人是“自然流失”（发券也没用），有些人是“被说服才留下”。我们可以引入 Uplift Model (增益模型)，只预测那些“发了券才会留下，不发就会走”的用户（Persuadables），避免给“铁杆粉丝”发券浪费钱。
  3. 实时性：
    • 答：如果是游戏或直播，可能需要 Real-time Scoring（用户刚退出游戏 5 分钟，系统立刻判定高危并推送），这需要用到 Flink + Redis + 在线模型服务。

📝 总结背诵口诀

一定义（啥叫流失？看多久？）
二找数（行为、交易、防穿越）
三造征（RFM、看趋势、算斜率）
四选模（XGB/LGB 是首选，准确率是大陷阱）
五评估（AUC、Lift、SHAP 解释清）
六落地（分层运营、A/B 测、算 ROI）

按照这个逻辑回答，既展示了技术深度，又体现了业务思维，是满分答案。
