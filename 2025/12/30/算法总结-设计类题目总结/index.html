系统设计类题目（Design Problems） 非常考察数据结构功底

🛠️ 第一部分：设计类题目的“万能公式”

所有设计题，本质上都是 “场景需求” + “数据结构组合”。

1. 核心解题步骤 (SOP)

  1. 明确接口 (API)：先列出需要实现的函数（如 get, put, follow, postTweet）。
  2. 分析复杂度要求：
    • 通常要求 get/put 为 $O(1)$ 或 $O(\log N)$。
    • 如果要求 $O(1)$ 查找 $\rightarrow$ 必须用哈希表 (HashMap)。
    • 如果要求有序/最值 $\rightarrow$ 考虑 堆 (Heap) 或 平衡树 (TreeMap)。
    • 如果要求维持顺序/最近使用 $\rightarrow$ 考虑 链表 (LinkedList) 或 双端队列 (Deque)。
  3. 设计数据结构组合：
    • 单一结构不够用时，就组合！
    • 经典组合：HashMap + Doubly Linked List (LRU/LFU), HashMap + Heap (TopK), Stack + Stack (Queue).
  4. 处理边界与细节：
    • 容量满了怎么办？（淘汰策略）
    • Key 不存在怎么办？（返回 -1 或 null）
    • 重复元素怎么处理？（Set 去重 或 计数 Map）

2. 常用数据结构“武器库”

数据结构 核心特性 典型应用场景 搭配建议
HashMap $O(1)$ 查找/删除/插入 快速定位节点、去重、计数 必选。几乎每个设计题都要用它来做索引。
Doubly Linked List $O(1)$ 插入/删除节点 (已知节点指针) 维护顺序、LRU 淘汰、实现队列/栈 搭配 HashMap 实现 $O(1)$ 的 LRU/LFU。
Heap (Priority Queue) $O(1)$ 取最值, $O(\log N)$ 插入/删除 TopK 问题、定时任务、中位数 搭配 HashMap 实现“可删除任意元素的堆”。
Stack / Queue LIFO / FIFO 括号匹配、浏览器历史、滑动窗口 两个栈实现队列，两个队列实现栈。
Trie (字典树) 前缀匹配、字符串统计 自动补全、敏感词过滤、IP 路由 节点中可存 HashMap 或数组。
Segment Tree / Fenwick 区间查询、单点更新 区间和、区间最大值、动态排名 较难，通常用于高阶设计题。

💡 第二部分：四大经典设计模式深度解析

这是你提到的几个题目的核心逻辑，务必吃透。

1. LRU (Least Recently Used) 缓存

  • 需求：容量满时，淘汰最久未使用的数据。get/put 需 $O(1)$。
  • 核心矛盾：HashMap 查找快但无序；链表有序但查找慢。
  • 解决方案：HashMap + 双向链表。
    • HashMap：Key -> Node，实现 $O(1)$ 查找。
    • 双向链表：维护访问顺序。头结点 = 最近使用，尾结点 = 最久未使用。
    • 操作逻辑：
      • get: 查 Map $\rightarrow$ 找到节点 $\rightarrow$ 移到链表头。
      • put:
        • 存在：更新值 $\rightarrow$ 移到链表头。
        • 不存在：新建节点 $\rightarrow$ 插到链表头 $\rightarrow$ 放入 Map。
        • 若超限：删除链表尾节点 $\rightarrow$ 从 Map 移除。
  • 关键点：双向链表的 addFirst, removeNode, removeLast 操作必须熟练手写。

2. LFU (Least Frequently Used) 缓存

  • 需求：容量满时，淘汰使用频率最低的数据；若频率相同，淘汰最久未使用的。
  • 核心难点：不仅要维护频率，还要维护同频率下的时间顺序。
  • 解决方案：HashMap + HashMap + 双向链表 (或者 HashMap + TreeSet)。
    • 主 Map：Key -> Node (存值、当前频率)。
    • 频率 Map：freq -> DoublyLinkedList (该频率下的所有节点，按时间排序)。
    • 最小频率变量：minFreq (记录当前最小的频率，避免遍历)。
    • 操作逻辑：
      • get: 查主 Map $\rightarrow$ 节点频率 +1 $\rightarrow$ 从旧频率链表移除 $\rightarrow$ 加入新频率链表头 $\rightarrow$ 更新 minFreq。
      • put: 类似，若超限，找到 minFreq 对应的链表，删除尾部节点。
  • 关键点：频率变化时，节点在不同链表间的迁移逻辑。

3. 设计 Twitter (关注系统 + 信息流)

  • 需求：发推、关注/取关、获取最近 10 条推文（包含自己和关注的人）。
  • 核心难点：如何高效合并多个用户的推文流？
  • 解决方案：HashMap + 链表/列表 + 多路归并 (堆)。
    • 用户映射：userId -> Set<followeeId> (关注列表)。
    • 推文存储：userId -> LinkedList<Tweet> (每个用户的推文链表，按时间倒序)。
    • 全局时间戳：用一个递增整数模拟时间。
    • 获取信息流 (getNewsFeed)：
      • 收集自己和所有关注者的推文链表头。
      • 使用 最大堆 (Max Heap) 进行 多路归并。
      • 每次弹出堆顶（最新推文），并将该链表的下一个节点入堆。
      • 取前 10 个。
  • 关键点：多路归并的逻辑，以及关注关系的去重（Set）。

4. 最小栈 (Min Stack)

  • 需求：栈的基本操作 + $O(1)$ 时间获取最小值。
  • 核心难点：弹出最小值后，如何知道第二小的值？
  • 解决方案：双栈法 或 单栈存差值。
    • 双栈法 (推荐)：
      • dataStack: 存正常数据。
      • minStack: 存对应时刻的最小值。
      • push(x): dataStack 压入 x; minStack 压入 min(x, minStack.peek())。
      • pop(): 两个栈同时弹。
      • getMin(): 返回 minStack.peek()。
  • 关键点：理解辅助栈的作用——“空间换时间”，记录历史状态。

🚀 第四部分：备战建议与面试技巧

  1. 手写双向链表是基本功：

    • LRU/LFU 都依赖它。面试时不要依赖语言自带的 LinkedList（除非允许），最好能自己定义 Node 类并实现 add, remove, moveToHead 方法。这非常加分！
    • 定义：class Node { int key, val; Node prev, next; }

  2. 注重“一致性”维护：

    • 在设计题中，最容易出错的是两个数据结构不同步。
    • 例如：在 LRU 中，从链表删除了节点，却忘了从 HashMap 删除，或者反之。
    • 口诀：凡是涉及两个结构的操作，写完一行代码，马上问自己“另一个结构更新了吗？”

  3. 考虑扩展性 (Follow-up Questions)：

    • 面试官常问：“如果数据量太大，内存放不下怎么办？” (提示：分片/Sharding)
    • “如果是多线程环境，怎么保证线程安全？” (提示：ReentrantLock 或 ConcurrentHashMap，虽然 LC 不考，但要能说出来)
    • “如果要支持过期时间怎么办？” (提示：惰性删除 + 定时任务)

  4. 代码风格：

    • 将功能拆分成私有 helper 方法（如 addNode, removeNode, updateFreq），不要让 put 方法里有几十行逻辑。清晰的模块划分是高级工程师的标志。

🎯 你的下一步行动

建议按照 Level 1 -> Level 2 -> Level 3 的顺序刷题。

  • LRU 和 最小栈 必须做到 15 分钟内 Bug Free 写完。
  • LFU 和 Twitter 要重点理解 数据流转的逻辑，可以在纸上画图模拟。
  • 做完后，尝试自己给题目加条件（比如：给 LRU 加上过期时间），思考怎么改。

这套设计题专题攻克下来，你在面对大厂的系统设计初阶问题和复杂数据结构题时，绝对能游刃有余！加油！
