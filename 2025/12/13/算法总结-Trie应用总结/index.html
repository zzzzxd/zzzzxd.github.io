🌳 一、Trie 树通用模板 (Java 实现)

这份模板采用了数组实现子节点（适用于小写字母），这是面试中最常用、效率最高的方式。如果是包含数字或特殊字符，可以改用 HashMap。

💡 进阶变体：如果字符集很大怎么办？

如果题目中包含大写字母、数字、甚至 Unicode 字符，new Node[26] 就不够用了。
修改方案：将 Node[] children 改为 Map<Character, Node> children。

  • 优点：支持任意字符，节省空间（稀疏图）。
  • 缺点：查询速度稍慢（从 $O(1)$ 变为 $O(\log K)$ 或均摊 $O(1)$），代码略繁琐。
  • 面试策略：先写数组版（默认小写），面试官问起扩展性时，口头说明“如果是全字符集，我会把数组改成 HashMap”。

🔍 二、如何识别要用 Trie 树？ (核心信号)

在面试中看到以下关键词或场景，请立刻条件反射想到 Trie 树：

信号关键词 含义解析 典型例子
“前缀” (Prefix) 题目明确问“是否有以…开头的单词”或“统计…开头的单词个数”。 208. 实现 Trie, 14. 最长公共前缀 (虽可用其他解法，但 Trie 最直观)
“自动补全” (Auto-complete) 输入一部分字符，推荐可能的完整单词。 677. 键值映射, 211. 添加与搜索单词
“字典” / “单词表” 给定一个庞大的单词列表，需要频繁查询某个词是否存在，或者查找变换一个字母的词。 212. 单词搜索 II (核心优化手段)
“异或最大值” (高阶信号) 涉及数字的二进制前缀匹配，求最大异或值。 421. 数组中两个数的最大异或值 (数字 Trie 树)
“敏感词过滤” 判断一段文本中是否包含列表中的任何敏感词。 实际业务场景，类似多模式匹配 (AC 自动机的前置知识)

❌ 什么时候 NOT 用 Trie？

  • 如果只需要简单的精确匹配（查某个词在不在），且数据量不大，HashSet 更快更简单。
  • 如果需要模糊匹配（如正则表达式 .* 在中间），Trie 处理起来很麻烦，可能需要结合 DFS/回溯。
  • 如果内存极其受限，Trie 的空间开销（每个节点一个数组）可能过大。

📝 三、高频题目场景与解题套路

场景 1：基础设计与前缀查询

  • 题目：208. 实现 Trie (Prefix Tree)
  • 套路：直接套用上面的模板。注意 search 和 startsWith 的区别（是否检查 isEnd）。

场景 2：Trie + 回溯 (DFS) —— 二维网格搜索

  • 题目：212. 单词搜索 II (字节/腾讯高频 Hard)
  • 场景：在一个字符矩阵中，找出所有存在于字典中的单词。
  • 痛点：如果对字典里每个单词都做一次 DFS，会超时。
  • Trie 解法：
    1. 把所有单词建成一棵 Trie 树。
    2. 在矩阵上进行 DFS。
    3. 剪枝优化：DFS 过程中，每走一步，就去 Trie 树里查一下“当前路径是否是某个单词的前缀”。
      • 如果 Trie 里没这个前缀 $\rightarrow$ 立即停止这条路（强力剪枝）。
      • 如果走到了 isEnd 节点 $\rightarrow$ 找到一个答案，记录下来（注意去重）。
  • 核心思想：用 Trie 树指导 DFS 的方向，避免无效搜索。

场景 3：Trie + 通配符搜索

  • 题目：211. 添加与搜索单词 - 数据结构设计
  • 场景：支持搜索包含 . 的单词，. 可以匹配任意字母。
  • 套路：
    • addWord: 普通插入。
    • search: 递归 DFS。
      • 遇到普通字符：像往常一样走 Trie 分支。
      • 遇到 .：遍历当前节点的所有非空子节点，只要有一条路能走通，就返回 true。

场景 4：数字 Trie (二进制位运算)

  • 题目：421. 数组中两个数的最大异或值
  • 场景：在一堆数字里找两个数，使它们的异或值最大。
  • 套路：
    • 把数字看作31 位的二进制字符串（从高位到低位）。
    • 构建一棵二进制 Trie 树。
    • 对于每个数，在 Trie 里贪心地找“相反”的路径（想要异或大，1 要找 0，0 要找 1）。
    • 这是 Trie 树在数值计算领域的经典应用。

场景 5：统计前缀出现次数

  • 题目：677. 键值映射 或自定义统计题
  • 场景：插入带权值的单词，查询某前缀下所有单词的权值和。
  • 套路：
    • 在 Node 类中增加一个字段 int val 或 int count。
    • 插入时，沿途经过的所有节点的 count 都 +1（表示有多少个单词经过这里）。
    • 查询时，走到前缀终点，直接返回该节点的 count 即可（$O(M)$ 完成统计，无需遍历子树）。

🚀 四、面试避坑指南

  1. 空间复杂度预警：

    • Trie 树非常耗内存。如果单词量极大（如百万级）且字符集大，面试官可能会问“如何优化空间？”。
    • 回答思路：提到可以使用 HashMap 代替数组，或者使用双数组 Trie (Double-Array Trie)（高级优化，提一下名字即可），或者压缩路径（Radix Tree / Patricia Tree）。

  2. 删除操作很难：

    • 标准的 Trie 模板通常不实现 delete。因为删除一个单词后，需要判断是否要删除整条路径（如果该路径不被其他单词共享）。
    • 如果面试官非要你写 delete，记得递归向上检查：如果子节点被删了，且当前节点不是其他单词的结尾 (isEnd=false) 且没有其他子节点，则删除当前节点。

  3. 大小写问题：

    • 一定要问清楚：“输入的字符串只包含小写字母吗？”
    • 如果有大写，要么转小写，要么开 52 大小的数组，要么直接用 HashMap。

总结

Trie 树 = 空间换时间的前缀匹配神器。
看到“前缀”、“补全”、“单词表匹配”，先想 HashSet 能不能做（只能精确匹配），如果不能，Trie 树就是标准答案。配合 DFS 使用，更是解决复杂字符串搜索的大杀器。

拿着这个模板和思路，去刷一下 [212. 单词搜索 II] 和 [211. 添加与搜索单词]，你对 Trie 的理解会瞬间升华！
